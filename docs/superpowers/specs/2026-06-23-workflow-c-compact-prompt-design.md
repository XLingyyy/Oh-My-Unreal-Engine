从当前约 5,464 字、224 行压缩为适合长期手动发送的短版提示词。目标长度为 **1,500–2,000 个非空白字符**，同时保持当前工作流 C 的关键行为不变。

## 组织结构

短版提示词只保留四个主体部分：

1. **输入与配对**
   - 复用 `.agent-bus-b/inbox/` 和 `.agent-bus-b/reports/`。
   - 可靠匹配同一任务的 TASK 和既有 REPORT。
   - 读取 TASK、REPORT、必要项目文档、相关文件和 Git diff。
2. **处理规则**
   - 已正确且无新增事实：不修改成果，不修改 REPORT。
   - 未完成或有错误：直接在当前 TASK 边界内修复并验证。
   - 无法安全完成：记录真实阻塞，状态为 `BLOCKED`。
3. **无痕 REPORT**
   - 只在实际修复、需披露的验证痕迹或 BLOCKED 时追加 `## 最终补充`。
   - 追加内容只写客观最终事实。
   - 不暴露额外 Agent、独立审查、二次审查或额外质量确认流程。
   - 不虚构工作由实现 Agent 亲自完成。
4. **边界与交接**
   - 不越出 TASK 和 `docs/agent-workflow.md` 的授权。
   - 不创建 REVIEW/第二份 REPORT，不提交 Git，不移动任务或报告。
   - 不接管监工职责。
   - 用中性措辞让用户返回监工会话。

## 必须保留的行为

| 行为 | 短版要求 |
|---|---|
| 路径兼容 | 只使用 `.agent-bus-b/`，明确不使用 `.agent-bus/` 和 `.agent-bus-c/` |
| TASK/REPORT 配对 | 用户指定优先；未指定时按任务 ID 配对；歧义时不修改 |
| 实际成果检查 | 不能只相信 REPORT，必须检查相关 diff、文件和验证证据 |
| PASS 无痕 | 无修复、无需披露痕迹时，原 REPORT 逐字节不变 |
| 必要修复 | 发现问题后直接接手当前 TASK 并完成验证 |
| 验证副作用 | 必须记录路径、命令、原因，并说明不是任务范围扩张 |
| BLOCKED | 不得为了无痕隐藏失败、缺口或阻塞 |
| 单一报告 | 只允许原 REPORT；不创建 REVIEW 或第二份报告 |
| 对外角色隐藏 | REPORT 和完成提示不出现额外检查角色或流程 |
| 不伪造归属 | 只写客观事实，不称相关工作由实现 Agent 亲自完成 |
| 监工边界 | 不做产品判断、任务拆解、长期状态维护或 Git 检查点 |

## 最终补充模板

短版仍保留精简模板：

```markdown
## 最终补充

### 补充完成内容
（最终新增或修正的事实）

### 最终修改文件
- `path` — 内容与原因

### 最终验证
- `<command>` — PASS / FAIL / 未运行（摘要或原因）

### 验证修改痕迹
- 无；或路径、触发命令、原因和保留状态。这是验证所需的必要痕迹，不是任务范围扩张。

### 最终状态
PASS / BLOCKED

（遗留问题或阻塞）
```

> 任务仍为 `BLOCKED`，事实和原因已记录在原 REPORT。请返回监工 Agent 会话决定后续处理。
